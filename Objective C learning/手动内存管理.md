对于面向对象的编程语言而言，程序需要不断地创建对象。开始创建时，程序创建的所有对象通常都有指针指向它，程序可能需要访问这些对象的实例变量或调用这些对象的方法，总之，这些对象是有用的。随着时间的流逝，程序再次创建了一些新的对象，而那些老的对象已经不会再被调用，也不再有指针指向它们，但如果程序没有回收它们占用的内存，就会出现内存泄漏。

如果内存一直泄露, 那么可用内存就会越来越少, 知道没有足够的内存去做更多的事, 应用程序可能会崩溃

对于有效的内存管理，通常认为包括两方面内容。

1. 内存分配:当程序创建对象时需要为对象分配内存。采用合理的设计，尽量减少对象的创建，并减少创建过程中的内存开销。
2. 内存回收:当程序不再需要对象时，系统必须及时回收这些对象所占用的内存，以便程序可以再次使用这些内存。

一般来说，内存的分配工作对程序影响小些，即使程序在一段时间内创建了过多的对象，造成了较大的内存开销，但是只要这些对象占用的内存得到了及时回收，程序也依然可以正常运行。而且，内存的分配操作相对比较容易，当程序创建对象时，系统会自动为这些对象分配内存。相比之下，内存的回收操作就显得更加复杂，管理起来也更加困难，为此，人们开发了多种内存回收策略，典型的内存回收策略有两种:

①自动回收，在这种策略下，系统会自动跟踪所有的对象，并在这些对象失去作用时回收它们占用的内存:

②混合回收，在这种策略下，系统会做一些工作，但同时也需要程序员在对象失去作用时通知系统。

Xcode 4.2引入了一个重要的新特性:自动引用计数(Automatic Reference Couting,ARC)。通过启用ARC特性，程序员不再需要重点关注内存回收相关的内容。对于i0S开发者而言，ARC是一个非常不错的东西，它大大降低了程序员进入i0S 应用开发的门槛。前面介绍的所有程序中，都没有主动去释放对象所占用的内存，只是在编译程序时使用了-fobjc-arc选项，这个选项就是启用Objective-C的 ARC 机制，这样ARC机制将会自动释放对象所占用的内存。
除此之外，如果程序员只是开发Mac应用程序，那么甚至可以使用自动垃圾回收机制，这种机制更加方便。
总结起来，Objective-C程序员可用的内存回收机制有如下3种。

1. 手动引用计数和自动释放池。
2. 自动引用计数(ARC)。
3. 自动垃圾回收。

## 对象的引用计数 

程序创建一个对象时，可能在一段时间内访问该对象的实例变量，也可能在这段时间内调用该对象的方法。当程序不再需要该对象时，就希望系统及时回收该对象所占用的内存。现在的问题是:系统怎么知道何时需要回收该对象呢?

Objective-C采用了一种被称为引用计数(Reference Counting)的机制来跟踪对象的状态:每个对象都有一个与之关联的整数，这个整数被称为引用计数。在正常情况下，当一段代码需要使用某个对象时，应将该对象的引用计数加1;当这段代码不再需要该对象时，应将该对象的引用计数减1，表示这段代码不再访问该对象。当对象的引用计数为0时，表明没有任何程序需要该对象，系统就会回收该对象所占用的内存。
系统在销毁该对象之前，会自动调用该对象的dealloc方法(该方法继承自NSObiect，因此所有的对象都有这个方法)来执行一些回收操作。如果该对象还持有其他对象的引用，此时必须重写dealloc方法，在该方法中释放该对象所持有的其他对象(通常就是调用被持有对象的release 方法将引用计数减1)。

当对象被销毁之后(它的引用计数变为 0, 并且系统自动调用了该对象的 dealloc 方法), 此时该对象已经不再存在; 如果有一个指针指向这个被销毁的对象, 这个指针就被称为悬空指针 (Dangling Pointer), 调用悬空指针指向对象的方法时, 程序往往会出现未知结果,甚至导致程序崩溃. 

手动引用计数回收内存的标准是: **当一个对象的引用计数为 0 时, 就表明程序不在需要该对象了, 从而通知系统回收该对象所占用的内存.** 

1. 在程序调用方法名为以 alloc , new, copy, multableCopy 开头的方法来创建对象时, 该对象的引用计数加一

> 上面所说的方法名都是指以驼峰写法 (每个单词首字母大写, 单词与单词之间无须任何分隔符)开头的, 比如, 名为 newBook 的方法会导致对象的引用计数加1, 但名为 newyork (字母y 没有大写) 的方法则不会导致对象的引用计数加一

2. 程序调用对象的retain方法时, 该对象的引用计数加一. 
3. 程序调用对象的release方法时, 该对象的引用计数减一

NSObject 中提供了有关引用计数的如下方法. 

- retain : 将该对象的引用计数加一 
- release : 将该对象的引用计数减一
- autorelease : 不改变该对象的引用计数的值, 只是将该对象添加到自动释放池中 
- retainCount : 返回该对象的引用计数的值

  

```objective-c
#import<Foundation/Foundation.h>

@interface FKItem : NSObject
- (id) init;
- (void) dealloc;
@end

@implementation FKItem
- (id) init {
    if (self == [super init]) {
        NSLog(@"init方法中, 引用计数为: %ld", self.retainCount);
    }
    return self;
}
- (void) dealloc {
    NSLog(@"销毁");
    [super dealloc];
}
@end
```

重写 dealloc 方法时一定要调用父类的 dealloc 方法,因为它的父类可能也有一些东西需要释放，执行[super dealloc]方法就是保证父类的 deallloc 方法被调用，从而保证父类对象持有的东西得到释放:而当前类重写dealloc方法时，先释放当前对象本身所占用的内存，接着调用[super dealloc]释放从父类继承得到的东西所占用的内存。

```objective-c
int main(int argc, const char * argv[]) {
    @autoreleasepool {
        // 调用 new 方法创建对象，该对象的引用计数为 1
        FKItem *item = [FKItem new];
        
        NSLog(@"初始引用计数: %ld", item.retainCount);
        
        [item retain]; // 引用计数加 1，retainCount 为 2
        NSLog(@"调用 retain 后: %ld", item.retainCount);
        
        [item retain]; // 引用计数加 1，retainCount 为 3
        NSLog(@"调用 retain 后: %ld", item.retainCount);
        
        [item release]; // 引用计数减 1，retainCount 为 2
        NSLog(@"调用 release 后: %ld", item.retainCount);
        
        [item retain]; // 引用计数加 1，retainCount 为 3
        NSLog(@"调用 retain 后: %ld", item.retainCount);
        
        [item release]; // 引用计数减 1，retainCount 为 2
        NSLog(@"调用 release 后: %ld", item.retainCount);
        
        [item release]; // 引用计数减 1，retainCount 为 1
        NSLog(@"调用 release 后: %ld", item.retainCount);
        
        NSLog(@"即将执行最后一次 release...");
        [item release]; // 引用计数减 1，retainCount 为 0，触发 dealloc
        
        // 注意：此处之后 item 已成野指针，严禁再调用 item.retainCount，否则会崩溃
        // NSLog(@"%ld", item.retainCount); // 这行代码会导致程序崩溃，请勿启用
    }
    return 0;
}

```

从程序的运行结果来看，每次调用对象的retain方法时，该对象的引用计数加1:每次调用对象的release 方法时，该对象的引用计数减1。当对象的引用计数变成0时，系统自动调用了该对象的 dealloc方法，接下来该对象已经被销毁了，程序不应该再调用该对象的方法了。

## 对象所属权

从上面的介绍可以看出，系统回收 Objective-C 对象的标准就是引用计数，当对象的引用计数为 0 时，该对象就会被系统回收。接下来设想如下场景：

➢ 在 main () 函数中创建一个 FKItem 对象。

➢ 程序中 FKUser 对象持有一个 FKItem 对象，程序可调用 FKUser 的 setItem: 方法来设置该 FKUser 持有 FKItem 对象。

此时的问题是：该 FKItem 对象属于 FKUser 还是属于 main () 函数？应该由 main () 函数来释放 FKItem 对象还是由 FKUser 来释放？

如果让 main () 函数来销毁 FKItem 对象，那么接下来 FKUser 对象所持有的 FKItem 对象其实被销毁了，因此，FKUser 对象中指向 FKItem 对象的指针就变成了悬空指针。

如果在 FKUser 对象的 dealloc 方法中就销毁了 FKItem 对象，那么可能出现的情况是：当 FKUser 的引用计数变为 0 时，FKUser 的 dealloc 方法被调用，这样它所持有的 FKItem 也被销毁了。接下来 main () 函数中指向该对象的指针就变成了悬空指针，这也是危险的。

由此可见，上面两种思路都不可行，常用的做法是：在 setItem: 方法中将传入 FKItem 对象的引用计数加 1，这样就可保证：只有当 main () 函数调用了 FKItem 的 release 方法将其引用计数减 1，且 FKUser 的 alloc 方法中再次调用 FKItem 的 release 方法将其引用计数减 1（二者不分先后）时，FKItem 对象的引用计数为 0，系统才会真正释放该 FKItem 对象。

```objective-c
#import<Foundation/Foundation.h>
@interface FKItem : NSObject

@end

@interface FKUser : NSObject {
    // 定义FKUser对象持有的FKItem对象
    FKItem* _item;
}
- (void) setItem: (FKItem*) item;
- (FKItem*) item;
@end

@implementation FKItem

@end

@implementation FKUser
- (void) setItem: (FKItem*) item {
    if (_item != item) {
        // 让item 的引用计数加一
        [item retain];
        _item = item;
    }
}
- (FKItem*) item {
    return _item;
}
- (void) dealloc {
    // 让_item的引用计数减一
    [_item release];
    NSLog(@"开始销毁 FKUser 对象");
    [super dealloc];
}
@end
```

当程序把 item 参数传给 setItem: 方法时，该方法执行时将会导致 FKItem 的引用计数加 1。 

在上面的 setter 方法中，先判断 _item 与被传入的 FKItem 对象是否相等也是很有必要的。如果不进行这个判断，当程序多次将同一个 FKItem 对象传给该 setter 方法时，将会导致每执行一次 setter 方法，被传入的 FKItem 对象的引用计数都加 1，这显然不是希望看到的结果。

程序重写了 FKUser 的 dealloc 方法，并在该方法中调用了 _item 的 release 方法将该对象的引用计数减 1，这样可保证与 setter 方法中对 _item 引用计数加 1 相抵。

其实手动内存释放的基本思路很简单，基本应该遵循：谁（包括对象、函数等）把对象的引用计数加了 1，谁就要负责在 “临死” 前把该对象的引用计数减 1。也就是说，任何实体（包括对象、函数）在结束前都应该把其他对象的引用计数恢复到开始前的状态。

接下来的程序只有当 main () 函数调用了 FKItem 的 release 方法，并且 FKUser 对象被销毁时，FKItem 对象才会被销毁。

```objective-c
int main(int argc, char* argv[]) {
    // 调用 new 方法创建对象, 引用计数为1
    FKItem* item = [FKItem new];
    NSLog(@"创建出来的引用计数为 %ld", item.retainCount);
    FKUser* user = [[FKUser alloc] init];
    [user setItem: item];
    NSLog(@"被FKUser对象持有后引用计数为 %ld", item.retainCount);
    // main方法将item的引用计数减一, item的引用计数为1
    [item release];
    NSLog(@"调用 release 后的引用计数为 %ld", item.retainCount);
    // user的引用计数减一, user的引用计数为 0
    // 系统调用 user 的dealloc方法, 调用dealloc 方法时, 将会让 FKItem 的引用计数减一
    // 这样 item 的应用计数变为 0, 系统会执行 item的dealloc 方法进行销毁
    [user release];
    return 0;
}
```



输出: 

```objective-c
创建出来的引用计数为 1
被FKUser对象持有后引用计数为 2
调用 release 后的引用计数为 1
开始销毁 FKUser 对象
```

从上面的程序可以看出，程序调用 FKItem 的 release 方法之后，该 FKItem 对象不会被销毁，因为此时它的引用计数依然为 1，只有等到程序调用 [user release] 导致该 FKUser 的引用计数变为 0，系统自动调用 FKUser 的 dealloc 方法来销毁 FKUser 对象，并且在 FKUser 的 dealloc 方法中，程序把 item 的引用计数再次减 1，使 item 的引用计数变为 0 时，系统才会销毁 item 所引用的对象。

当系统销毁FKUser时, FKUser也被销毁了, 这样, 上面的程序不会导致内存泄露

## 方法中的保留和释放 

```objective-c
int main(int argc, char* argv[]) {
    // 调用 new 方法创建对象, 引用计数为1
    FKItem* item = [FKItem new];
    NSLog(@"创建出来的引用计数为 %ld", item.retainCount);
    FKUser* user = [[FKUser alloc] init];
    [user setItem: item];
    NSLog(@"被FKUser对象持有后引用计数为 %ld", item.retainCount); 
    // 重建一个新对象
    FKItem* item2 = [FKItem new]; 
    // 将item2作为参数传给setItem: 方法
    [user setItem: item2]; 
    // main方法将item的引用计数减一, item的引用计数为1
    [item release]; 
    // main方法将item2的引用计数加减一, item2的引用计数为 1
    [item2 release]; 
    NSLog(@"调用 release 后的引用计数为 %ld", item.retainCount);
    // user的引用计数减一, user的引用计数为 0
    // 系统调用 user 的dealloc方法, 调用dealloc 方法时, 将会让 FKItem 的引用计数减一
    // 这样 item 的应用计数变为 0, 系统会执行 item的dealloc 方法进行销毁
    [user release];
    return 0;
}
```

相比之前的程序，这个代码主要多了创建item2, 将 item2 作为参数传给 FKUser 的 setItem: 方法，此时的后果是：FKUser 原来持有的 FKItem 对象（由 item 指针所引用）的引用计数依然为 2，并没有减 1，即使后面程序调用了 item 的 release 方法让 item 的引用计数减 1，但 item 的引用计数依然为 1，而不是 0—— 这样 item 引用的对象就不会被释放了。

这个程序只回收了item2所指向的对象所占用的内存空间, 而item 所指向的对象没有被释放, 这就造成了内存泄露

为了保证item所指向的对象不会造成内存泄露, 必须保证在 setter 方法中将原来的_item实例变量所引起的retainCount减一, 再新传入的参数\_item加一, 也就是说, 需要将FKUser的setter: 方法修改为如下形式: 

```objective-c
- (void) setItem: (FKItem*) item {
    if (_item != item) {
        // 将_item引用的实例变量的引用计数减一
        [_item release];
        // 让item 的引用计数加一, 并赋给_item实例变量
        _item = [item retain]; 
    }
}
```

这样就可以保证在新传入的FKItem对象的引用计数加一之前, 先将FKUser对象原来持有的FKUser对象的引用计数减一, 这样就可以让item , item2两个指针所引用FKItem对象都被回收了

## 使用自动释放池

如果在函数、方法开始处将对象的引用计数加 1，接下来在函数、方法不需要该对象时就应该对该对象的引用计数减 1。这个理论基本正确，但是有些函数、方法需要返回一个对象，那么问题就产生了：该函数、方法需要返回的对象肯定不能立即将引用计数减 1，这样系统可能在该对象被返回之前，已经销毁了该对象。

为了保证函数、方法返回的对象不会在被返回之前就被销毁，就需要保证被函数、方法返回的对象能被延迟销毁。为了实现这种延迟销毁，有如下两种做法。

+ 程序每次获取并使用完其他函数、方法返回的对象之后，立即调用该对象的 release 方法将函数、方法返回对象的引用计数减 1。

+ 使用自动释放池进行延迟销毁。

  第一种方法

  ```objective-c
  FKItem* productItem(void) {
      FKItem* item = [FKItem new];  //引用计数为1
      NSLog(@"返回之前的引用计数为 %ld", item.retainCount);
      // 返回的对象的引用计数w为1
      return item;
  }
  int main(int atgc, char* argv[]) {
      // it的引用计数为 1
      FKItem* it = productItem();
      NSLog(@"%ld", it.retainCount);
      // ...
      // 程序执行完成后, 将 it(引用prodectItem()函数返回值)的引用计数减一
      // 程序使用productItem()函数返回值完成后, 延迟销毁 it 所指向的对象
      [it release];
      return 0;
  }
  ```

  从上面的程序可以看出，对于返回 FKItem 对象的 productItem () 函数而言，它只是负责创建 FKItem 对象，并将它的引用计数加 1，该函数并没有把它的引用计数减 1。

  接下来在 main () 函数中调用 productItem () 函数时，将会使用一个 FKItem 变量来保存 productItem () 函数的返回值，当该函数的返回值使用完成后，程序调用 [it release] 代码将 FKItem 对象的引用计数减 1，使它的引用计数变为 0，从而使系统销毁该对象。

上面的做法虽然简洁、易懂，但违背了前面介绍的原则：谁负责将对象的引用计数加 1，谁就应该负责将该对象的引用计数减 1—— 因此，这种做法并不是一种优雅的设计。



更好的做法是使用自动释放池来解决这个问题。所谓自动释放池，就是一个存放对象的容器（比如集合）。自动释放池会保证延迟释放该池中所有的对象。

出于自动释放的考虑，所有的对象都应该添加到自动释放池中，这样可以让自动释放池在销毁之前，先销毁池中的所有对象。把一个对象添加到自动释放池中，可调用该对象的 autorelease 方法，其签名如下。

> `- (id) autorelease：`该方法不会修改对象的引用计数，只是将该对象添加到自动释放池中。该方法将会返回调用该方法的对象本身。

由于 autorelease 方法预设了一个在将来某个时间将要调用的 release 方法，该方法会把对象的引用计数减 1—— 到底在将来的哪个时间呢？取决于自动释放池的释放时间，当程序在自动释放池中下文调用某个对象的 autorelease 方法时，该方法只是将该对象添加到自动释放池中，当该自动释放池释放时，自动释放池会让池中所有的对象执行 release 方法。

借助自动释放池的功能，可以控制当函数、方法需要返回一个对象时，先调用该对象的 autorelease 方法。这样只要在自动释放池上下文中调用该方法时，这个对象就会被添加到自动释放池中。程序只要控制自动释放池的销毁即可。

上面的代码如果使用自动释放池, 即可将productItem() 函数的代码改为如下: 

```objective-c
FKItem* producItem(void) {
    FKItem* item = [FKItem new];   // 引用计数为1
    // autorelease 不会改变对象的引用计数
    // 但程序执行autorelease 方法时, 会将该对象添加到自动释放池中
    return [item autorelease];
}
```

## 自动释放池的销毁时机与工作过程

自动释放池的销毁时机与其他普通对象的销毁时机相同, 只要自动释放池中的引用计数为 0, 系统就会自动销毁自动释放池的对象.从这个意义上可以看出, 自动释放池并没有任何特别之处, 它也只是一个Objective-C对象, 同样遵守相同的对象回收机制 

创建和销毁自动释放池的代码模板如下: 

```objective-c
NSAutoreleasePool* pool = [[NSAutoreleasePool alloc] init]; 
... 
// 自动释放池的引用计数减一
[pool release]; 
```

程序创建 NSAutoreleasePool 对象之后，该对象的引用计数为 1，接下来当程序调用该对象的 release 方法后，该对象的引用计数变为 0，系统准备调用 NSAutoreleasePool 的 dealloc 方法来回收该对象，系统会在调用 NSAutoreleasePool 的 dealloc 方法时回收该池中所有的对象。

> NSAutoreleasePool 还提供了一个 drain 方法来销毁自动释放池的对象。与 release 不同的是，release 会导致自动释放池自身的引用计数变为 0，从而让系统回收 NSAutoreleasePool 对象，在回收 NSAutoreleasePool 对象之前，系统会回收该池中所有的对象。而 drain 方法则只是回收池中所有的对象，并不回收自动释放池本身。

```objective-c
int main(int argc, char * argv[]) {
    @autoreleasepool {
        NSAutoreleasePool* pool = [[NSAutoreleasePool alloc] init];
        // it的引用计数为1
        FKItem* it = productItem();
        // 接下来可以调用it的方法
        NSLog(@"%ld", it.retainCount);
        // ...
        // 创建一个FKUser对象，并将它添加到自动释放池中
        FKUser* user = [[[FKUser alloc] init] autorelease];
        // 接下来可以调用user的方法
        NSLog(@"%ld", user.retainCount);
        // ...
        // 系统将因为池的引用计数变为0而回收自动释放池
        // 回收自动释放池时会调用池中所有对象的release方法
        [pool release];
    }
    return 0;
}
```

在上面程序中，第一行粗体字代码创建了一个 NSAutoreleasePool 对象，最后一行粗体字代码将自动释放池的引用计数减 1，系统将因为池的引用计数变为 0 而回收自动释放池，回收自动释放池时会调用池中所有对象的 release 方法。

上面程序在自动释放池上下文中一共得到了两个对象，其中 FKItem 对象通过 productItem () 函数获得，该函数最后一行代码在返回 FKItem 对象时调用了 autorelease 方法，该方法会把该 FKItem 对象添加到自动释放池中；而 FKUser 对象则在创建完成后立即调用 autorelease 方法，该方法也会把该 FKUser 对象添加到自动释放池中。

使用自动释放池之后, 程序只要把所有通过以 alloc, new, copy. mutableCopy等开头的方法创建出来的对象添加到自动释放池中, 接下来程序只要控制释放自动释放池即可, 自动释放池就会负责把容器中所有对象的引用计数减一

## 临时对象与事件循环中的自动释放池

对于Foundation 中的类而言, 当调用方法创建对象时, 只要这些方法不是以 alloc, new, copy, mutableCopy 等开头的, 系统都会默认创建自动释放的对象. 

```objective-c
NSMutableArray* array = [[NSMutableArray alloc] init];  // 引用计数为1 
// 接下来使用该 array 
// ... 
[array release]; 
```

上面的代码使用的alloc方法创建一个数组, 因此通过这种方法创建的对象不是自动释放的对象, 如果显示将它们添加到自动释放池中, 程序必须手动调用它们的 release 方法将它们的引用计数减一

如果改为使用以下代码: 

```objective-c
NSMutableArray* array = [NSMutableArray arrayWithCapacity: 20]; 
// 接下来使用该 array 
// ...
```

使用arrayWithCapacity: 方法创建一个NSMutableArray 对象, 由于创建该对象的方法不是以 alloc , new, copy, mutableCopy开头的, 因此该方法默认创建了一个自动释放的对象. 当自动释放池时, 该对象就会被自动释放, 因此这种对象也被称为临时对象. 

### 事件循环

事件循环指的是当程序中某个组件事件被激发, (比如按钮被单击等)后, 系统将会回调某个方法来处理该事件, 注意: 事件循环并非这么简单. 

+ 创建自动释放池 
+ 调用事件处理方法
+ 销毁自动释放池 
+ 处理完成, 方法返回 

事件循环会自动创建自动释放池上下文, 因此在事件处理方法中创建临时对象都会被销毁. 如果程序希望在事件处理方法中对实例变量赋值, 那么就需要格外小心, 如果赋值对象只是临时对象, 那么对象赋值完成和事件处理结束时, 该对象已经被回收了

如果程序需要在事件处理结束后依然可以保留临时变量,则可以采用如下两种方式之一 

1. 在将临时对象赋值之前, 先调用临时对象的retain 方法将它的引用计数加一 
2. 把临时对象赋值给 retain, strong, 或 copy 指示符修饰的属性(**注意: assign, weak, unsafe_unretain指示符修饰的属性除外**). 

 对于第一种方法, 既然是一个临时对象, 可能会被自动释放池所回收, 那么先调用该对象的retain方法给它的引用计数加一即可. 

对于第二种方法: 例如

```objective-c
@property (nonatomic, retain) NSMutableArray* data; 
@end 
```

在上面的程序中，data 属性使用了 retain 指示符（只有当未启用 ARC 机制时，该指示符才有效），表明系统在将被赋值对象赋给该属性之前，会先将被赋值对象的引用计数加 1。使用 retain 指示符修饰的属性对应的 setter 方法如下

```objective-c
if (property != newValue) {
    [property release]; 
    property = [newValue retain]; 
}
```

其中的data属性使用copy指示符也是可以的, 使用 copy 指示符修饰的属性对应的 setter 方法如下: 

```objective-c
if (property != newValue) {
    [property = release]; 
    property = [proValue copy]; 
}
```

上面的做法虽然没有把newValue的引用计数加一, 但如果被赋值的newValue只是一个临时对象, 当自动释放池释放时, newValue 将会被释放, 上面的代码返回了newValue 的副本, 而且copy 方法会让副本的引用计数加一, 因此这种方式虽然没有把临时对象本身保留下来, 但保留了临时对象的副本. 

对于这种方式, 由于MyView类使用data保留了NSMutableArray对象, (将该对象的引用计数加了1), 因此程序需要在MyView的dealloc方法中将该 NSMutableArry 对象的引用计数减一, 也就是常重写MyView 类方法 

```objective-c
- (void) dealloc {
    [data release]; 
    [super dealloc]; 
}
```

## 手动内存管理的规则总结

下面给出有关手动内存管理的规则。

➤ 调用对象的 release 方法并不是销毁该对象，只是将该对象的引用计数减 1；当一个对象的引用计数为 0 时，系统会自动调用该对象的 dealloc 方法来销毁该对象。

➤ 当自动释放池被回收时，自动释放池会依次调用池中每个对象的 release 方法。如果该对象调用 release 方法后引用计数变为 0，那么该对象将要被销毁；否则该对象可以从自动释放池中 “活” 下来。

➤ 当程序使用以 alloc、new、copy、mutableCopy 开头的方法创建对象后，该对象的引用计数为 1，当不再使用该对象时，需要调用该对象的 release 方法或者 autorelease 方法。

➤ 如果使用 retain 方法为对象增加过引用计数，则用完该对象后需要调用 release 方法来减少该对象的引用计数，并保证 retain 次数与 release 次数相等。

➤ 如果通过其他方法获取了对象，且该对象是一个临时对象，若在自动释放池上下文中使用该对象，那么使用完成后无须理会该对象的回收，系统会自动回收该对象。如果程序需要保留这个临时对象，则需要手动调用 retain 来增加该临时对象的引用计数；或者将该临时对象赋值给 retain、strong 或 copy 指示符修饰的属性。

➤ 在 Cocoa 或 iOS 的事件循环中，在每个事件处理方法执行之前都会创建自动释放池，方法执行完成后会回收自动释放池。如果希望自动释放池被回收后，池中某些对象能 “活” 下来，程序必须增加该对象的引用计数，保证该对象的引用计数大于它调用 autorelease 的次数。

> **autorelease 的本质是「一次延迟的 release」释放池回收时，会对池内所有标记了 `autorelease` 的对象执行 `release`，这会让对象的引用计数 **减 1**。如果此时对象的引用计数 ≤ 0，就会被销毁；如果想让对象「活下来」，必须保证：对象当前的引用计数 > autorelease 带来的 release 次数（通常是 1）**

## 自动引用计数概述

自动引用计数（ARC）可以避免手动引用计数可能引入的一些潜在陷阱，自动引用计数同样是基于引用计数的，只不过由系统来负责管理对象的引用计数，系统可以判断何时需要保持对象，何时需要自动释放对象。

开发者不需要担心方法内创建的对象，编译器会管理好对象的内存，通过生成正确的代码去释放或保持返回的对象。

## @autoreleasepool块

@autoreleasepool 块定义自动释放池上下文，任何在该上下文中创建的对象都由 ARC 来自动释放，并在自动释放块结束时销毁这些对象。

如果程序在执行过程中创建了大量的临时对象（比如在循环中创建对象，这是一种非常差的做法），程序可能需要多次释放这些临时对象，此时程序可以考虑将 @autoreleasepool 块放在循环体内，

```objective-c
for(int i = 0 ; i < 100 ; i++)
{
    @autoreleasepool {
        // 创建临时对象
        // 调用临时对象的方法
        // ...
    }
}
```

上述代码在循环体每次执行完成时，都会销毁这些临时对象，从而避免整个循环结束时才去销毁这些临时对象，这样也便于更好地管理内存。
